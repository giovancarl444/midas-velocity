3 next.config.js Changes That Cut My Vercel Bill 70%

Published May 2026 · Cost Optimization

Three lines. Not joking. My side project went from $400/month to $80/month on Vercel. No migration. No rewrites. Just config.

Here are the exact changes, why they work, and the edge cases to watch for.

1. Pin Your Function Region

Vercel defaults your serverless functions to a region. If your users are in New York and your functions run in San Francisco, every request pays a latency tax and a cold-start tax.

// vercel.json { "regions": ["iad1"] }

Common region codes:

  • iad1 — Washington, D.C. (US East)
  • sfo1 — San Francisco (US West)
  • fra1 — Frankfurt (EU Central)
  • hkg1 — Hong Kong (Asia)
  • syd1 — Sydney (Oceania)

Edge case: If you have users globally and one database, pin to the database region. Don't try to multi-region unless you're on Enterprise and know what you're doing.

2. Static Export Where Possible

Next.js defaults to SSR for everything. That's great for flexibility. It's terrible for your bill.

// next.config.js module.exports = { output: 'export' }

Static pages are served from Vercel's edge cache. That means:

  • Zero function invocations = zero function billing
  • Sub-100ms response times globally
  • No cold starts ever

Edge case: You can't use API routes with static export. For hybrid apps, use generateStaticParams per route instead of global export.

3. Force AVIF + Long Cache

Image optimization is Vercel's hidden cost killer. Bandwidth bills scale with image size. AVIF is 40-60% smaller than JPEG.

// next.config.js images: { formats: ['image/avif', 'image/webp'], minimumCacheTTL: 60 * 60 * 24 * 30 }

The minimumCacheTTL line is the real hack. Without it, Vercel re-optimizes the same image on every cache miss. With it, optimized images are cached for 30 days. On a high-traffic site, that's thousands of avoided optimizations.

Edge case: AVIF encoding is slower than WebP. If you have real-time image uploads, WebP-only may be safer for user experience.

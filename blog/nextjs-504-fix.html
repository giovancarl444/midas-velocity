Fix Next.js 504 Errors on Vercel

Published May 2026 · Deployment Engineering

504 Gateway Timeout is the most common production failure I see on Vercel. It doesn't happen on localhost. It only hits when real users show up. And when it hits, it hits at the worst possible moment — launch day, campaign traffic, or that Hacker News spike you've been waiting for.

Here's the diagnostic flow I use to fix it in under an hour.

Step 1: Check Function Duration

Vercel's default serverless function timeout is 10 seconds on Hobby, 60 seconds on Pro, and 300 seconds on Enterprise. If your API route or SSR page takes longer, Vercel kills it. 504.

// Check in Vercel Dashboard → Functions → [your function] → Duration // Or add logging: export default async function handler(req, res) { const start = Date.now(); // ... your logic console.log(`Duration: ${Date.now() - start}ms`); res.json({ ok: true }); }

If you're consistently near the limit, you have three options:

  • Optimize the function (add caching, reduce DB queries)
  • Move to Edge Runtime (faster cold starts, but limited API)
  • Upgrade your plan (if you genuinely need >10s)

Step 2: Check Memory Limits

Default memory is 1024 MB. If your function exceeds this, Vercel kills it mid-execution. 504.

// Increase memory in vercel.json { "functions": { "api/heavy-task.js": { "maxDuration": 30, "memory": 3008 } } }

Step 3: Fix Cold Starts

A cold start on a large function can eat 3-5 seconds before your code even runs. In regions far from your database, add network latency and you're at 8-9 seconds with almost no time left for actual work.

Fixes:

  • Pin region to your database region (see The Vercel Tax)
  • Use Edge Runtime for simple transforms
  • Keep functions warm with cron triggers (if on Pro)
  • Minimize dependencies — bundle size directly impacts cold start time

Step 4: Database Connection Pooling

Serverless functions create new connections on every invocation. Without pooling, you'll exhaust your database's connection limit under load. The function hangs waiting for a connection. 504.

// Use a connection pooler like Prisma Accelerate or @neondatabase/serverless import { Pool } from '@neondatabase/serverless'; const pool = new Pool({ connectionString: process.env.DATABASE_URL }); export default async function handler(req, res) { const client = await pool.connect(); // ... query client.release(); }

The 504 Decision Tree

  1. Does it happen on every request or only under load? → Every = code bug. Under load = scaling limit.
  2. Does it happen on static pages or dynamic routes? → Static = CDN issue (rare). Dynamic = function issue.
  3. Does it correlate with database queries? → Check pooling and query time.
  4. Does it happen after deployments? → Cold start + first request = timeout.
